# 推奨言語モデル選択ガイド

このリポジトリでは、**次にやることが決まっているとき**、エージェントは実装着手前に推奨モデルを必ず提示する。  
第一目的は **Quota 節約**。ツール品質を落とさない範囲で、必要なときだけ上位モデルを使う。

Cursor ルール: `.cursor/rules/model-recommendation.mdc`（`alwaysApply: true`）

## 候補（4択のみ）

| モデル | 向く作業 |
|--------|----------|
| **Composer 2.5** | 設計・方針が固まった実装、UI改修、`events.json` 追記、extract 修正、小さなバグ修正 |
| **Grok 4.6 Low** | 軽い調査、ドキュメント文言、単純な件数確認、README レベルの整理 |
| **Grok 4.6 Medium** | 仕様が一部曖昧な改修、集計ロジックの切り分け、スキーマ変更＋UIの設計調整 |
| **Grok 4.6 High** | 実機と大きくズレる原因調査、要件の再設計、extract データ源の変更 |

## いつ提示するか

| タイミング | 提示 |
|------------|------|
| ユーザーが次タスクを指示した直後 | 必須 |
| 設計レビュー完了 → 実装フェーズ | 必須 |
| セッション終了時に「次は〇〇」と提案するとき | 推奨 |
| 質問のみ・方針未確定 | 不要 |

## このプロジェクトでの目安

### Composer 2.5 で足りる例

- 案1イベントUI（`auto` / `single`）の実装
- サポカ・ウマ娘セレクトのUI改善（検索・絞込など設計済み）
- 優先サポカの `events.json` 追記（イベント内容がユーザー提供済み）
- `npm run extract` 後の skillId 補完確認

### Medium 以上を検討する例

- `events.json` のスキーマ変更が集計・既存データと絡む
- ラジオ選択と `max(hintLv)` の整合が実機と合わない原因調査
- 複数ファイルにまたがるリファクタの設計判断

### High を検討する例

- 合計 SP が実機と大きくズレるが再現条件が不明
- extract のデータ源変更（`available_skill_set` 以外を足す等）
- 要件そのものの再設計

## 提示テンプレート

エージェントは次の形式で返す。

```markdown
## 推奨言語モデル
**Composer 2.5**（設計済みのため実装中心で足りる）

| モデル | このタスクでの位置づけ |
|--------|------------------------|
| Composer 2.5 | ◎ 推奨 |
| Grok 4.6 Low | △ 実装エージェントとしては不向き |
| Grok 4.6 Medium | △ 仕様ブレが出たら検討 |
| Grok 4.6 High | × 不要 |
```

## 人間向けメモ

- エージェントの推奨は**次チャットで使うモデル**の目安。現在のチャットのモデルは自動では切り替わらない。
- 実装後に実機とズレたら、1段階上のモデルで再調査する。
